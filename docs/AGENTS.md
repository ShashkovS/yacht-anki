# AGENTS.md — Инструкция по ведению разработки

## О проекте

Яхтенный тренажёр на базе интервального повторения (FSRS) для подготовки к гонкам на Fareast 28R.
Приложение показывает 2D-диаграммы яхтенных ситуаций (вид сверху) и задаёт вопросы по терминологии, манёврам и правилам расхождения.

## Стек

- **Backend**: Python 3.14, aiohttp, aiosqlite, yoyo-migrations
- **Frontend**: React 19, TypeScript, Vite, Tailwind CSS, react-konva, ts-fsrs
- **БД**: SQLite (STRICT tables, WAL mode)
- **Тесты**: pytest (backend), vitest (frontend unit), playwright (e2e)
- **Деплой**: Docker + docker-compose

## Ключевые архитектурные решения

- **Колоды глобальные** — одинаковые для всех пользователей, загружаются сидами
- **Нет UI-редактора карточек** — контент управляется разработчиком через JSON/сиды
- **FSRS на фронтенде** — вычисления через ts-fsrs, бэкенд только персистит состояние
- **Визуализация** — react-konva, 2D Canvas, вид сверху, процедурный рендер из DiagramSpec JSON
- **Язык интерфейса** — только русский
- **API** — POST-based JSON, конверт `{ok: true, data: ...}`

## Порядок работы

### Перед началом фазы
1. Прочитать документ фазы (`docs/phaseNN.md`)
2. Прочитать исследования (`docs/v1.md`, `docs/v2.md`) при работе с контентом или визуализацией
3. Убедиться, что предыдущая фаза завершена и `make test` проходит

### Во время работы
1. Следовать паттернам существующего кода:
   - SQL-запросы → `backend/db/*.py`
   - HTTP-маршруты → `backend/http/*_routes.py`
   - Страницы → `frontend/src/pages/`
   - Фичи → `frontend/src/features/`
   - Утилиты → `frontend/src/shared/`
2. Писать тесты параллельно с кодом
3. Каждый файл начинать с короткого комментария о назначении
4. Запускать `make test` после каждого логического блока

### После завершения фазы
1. Убедиться, что все критерии завершения из документа фазы выполнены
2. Запустить полный `make test`
3. Проверить работу в браузере (для UI-фаз)
4. Обновить документ фазы — отметить завершённые пункты, добавить заметки если что-то пошло не по плану

## Правила разработки

### Код
- Простой и явный код без "умных" абстракций
- SQL в `backend/db/`, маршруты в `backend/http/`
- Один модуль — одна ответственность
- Не добавлять фичи сверх описанных в текущей фазе

### Тесты
- Backend: pytest с async-фикстурами и тестовой БД
- Frontend unit: vitest с @testing-library/react
- E2E: playwright для кросс-браузерных сценариев
- Не удалять и не пропускать тесты. Если тест ломается — чинить

### Зависимости
- Python: `uv add` / `uv add --dev`
- Frontend: `npm install` / `npm install -D`
- Не редактировать lock-файлы вручную

### Контент карточек
- Источник терминов и правил: `docs/v1.md` и `docs/v2.md`
- DiagramSpec — JSON в сидах, не в коде компонентов
- Карточки загружаются через `backend/db/seed.py` при старте в dev-режиме

## Схема базы данных

### Существующие таблицы (не трогать)
- `users` — пользователи и авторизация
- `refresh_sessions` — JWT refresh-токены

### Новые таблицы
- `decks` — глобальные колоды (без user_id)
- `cards` — карточки с diagram_spec JSON
- `card_states` — прогресс пользователя (FSRS-состояние per card per user)
- `review_log` — история всех ответов
- `user_settings` — настройки повторения per user

Подробности миграций — в `docs/phase01.md`.
