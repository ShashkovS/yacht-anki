# Контентные JSON-файлы

В папке `content` лежат встроенные колоды приложения:

- `terms.json`
- `manoeuvres.json`
- `right-of-way.json`

Бэкенд загружает их в фиксированном порядке: `terms`, `manoeuvres`, `right-of-way`.
Имя файла и поле верхнего уровня `slug` должны совпадать. Если файл отсутствует, `slug` пустой или неправильный, сидинг встроенного контента падает с ошибкой.

## Общая схема файла колоды

Каждый JSON-файл должен содержать объект такого вида:

```json
{
  "slug": "terms",
  "title": "Термины и курсы",
  "description": "Короткое описание колоды.",
  "builtin": true,
  "cards": []
}
```

### Обязательные поля колоды

- `slug`: непустая строка.
- `title`: непустая строка.
- `description`: непустая строка.
- `builtin`: boolean. Для встроенного контента сейчас используется `true`.
- `cards`: массив карточек.

## Общая схема карточки

Каждый элемент в `cards` должен быть объектом такого вида:

```json
{
  "slug": "close-hauled",
  "template_type": "term_definition",
  "prompt": "Что за курс на этой схеме?",
  "answer": "Крутой бейдевинд",
  "explanation": "Это острый курс к ветру.",
  "diagram_spec": {},
  "tags": ["courses"],
  "sort_order": 2
}
```

### Обязательные поля карточки

- `slug`: непустая строка. Должен быть уникален внутри одной колоды.
- `template_type`: непустая строка с типом карточки.
- `prompt`: непустая строка.
- `answer`: непустая строка.
- `explanation`: непустая строка.
- `diagram_spec`: объект.
- `tags`: массив строк.
- `sort_order`: положительное целое число.

### Важные правила

- Два одинаковых `slug` в одной колоде запрещены.
- При повторном сидинге существующая карточка обновляется по паре `deck.slug + card.slug`.
- `sort_order` задаёт порядок карточек внутри колоды.
- На уровне бэкенда для `diagram_spec` проверяется только то, что это объект. Но для нормальной работы фронтенда структура должна соответствовать схеме ниже.

## Поддерживаемые `template_type`

Сейчас в контенте используются такие типы:

- `term_definition`: обычная карточка с вопросом и ответом по термину.
- `directional`: карточка, где пользователь должен повернуть лодку в правильный курс.
- `trim`: карточка с выбором одного варианта ответа.
- `concept`: карточка с нейтральным выбором одного варианта ответа по правилу или термину.
- `manoeuvre`: карточка, где после открытия показывается порядок шагов.
- `right_of_way`: карточка, где нужно выбрать лодку, которая должна сторониться.

### Какой `expected_answer` подходит к какому типу

- `term_definition`: обычно без `expected_answer`.
- `directional`: `expected_answer.type = "rotate_heading"`.
- `trim`: `expected_answer.type = "choose_option"`.
- `concept`: `expected_answer.type = "choose_option"`.
- `manoeuvre`: `expected_answer.type = "reveal_steps"`.
- `right_of_way`: `expected_answer.type = "select_boat"`.

Если тип ответа не совпадает с `template_type`, UI карточки не сможет показать нужное интерактивное поведение.

## Схема `diagram_spec`

`diagram_spec` описывает сцену диаграммы, а иногда и правильный ответ.

Минимальный рабочий вариант:

```json
{
  "version": 1,
  "wind": {
    "direction_deg": 0
  },
  "boats": [
    {
      "id": "alpha",
      "x": 500,
      "y": 350,
      "heading_deg": 45,
      "sails": {
        "main": {
          "state": "trimmed"
        }
      },
      "label": "A"
    }
  ]
}
```

### Поля верхнего уровня `diagram_spec`

- `version`: сейчас поддерживается только `1`. Если поле не указано, фронтенд всё равно ожидает версию `1`.
- `wind`: обязательный объект ветра.
- `boats`: обязательный непустой массив лодок.
- `mark`: необязательный объект знака.
- `overlays`: необязательные подсказки и служебные наложения для правил расхождения.
- `answer_scene`: необязательная отдельная сцена, которая показывается после открытия ответа.
- `expected_answer`: необязательное описание правильного ответа.

## Объект `wind`

```json
{
  "direction_deg": 0,
  "speed_knots": 12
}
```

- `direction_deg`: обязательное конечное число. Это направление, **откуда** дует ветер.
- `direction_deg = 0`: ветер приходит с верхнего края экрана.
- `speed_knots`: необязательное число. Поддерживается парсером, но в текущем контенте не используется.

## Массив `boats`

Каждая лодка:

```json
{
  "id": "alpha",
  "x": 500,
  "y": 350,
  "heading_deg": 45,
  "label": "A",
  "highlight": "answer",
  "sails": {
    "main": {
      "state": "trimmed"
    },
    "jib": {
      "state": "trimmed"
    },
    "gennaker": {
      "visible": true,
      "state": "trimmed"
    }
  }
}
```

### Обязательные поля лодки

- `id`: непустая строка.
- `x`: число, координата по горизонтали.
- `y`: число, координата по вертикали.
- `heading_deg`: число, курс лодки в градусах относительно сцены.
- `heading_deg = 0`: нос лодки смотрит вверх по экрану.
- Дальше углы растут по часовой стрелке относительно экрана.
- `sails.main`: объект грота.

### Необязательные поля лодки

- `label`: короткая подпись, обычно `A` или `B`.
- `highlight`: одно из значений:
  - `none`
  - `selected`
  - `answer`
- `sails.jib`: объект стакселя.
- `sails.gennaker`: объект генакера.

## Объекты парусов

Обычный парус:

```json
{
  "state": "trimmed",
  "angle_deg": 20
}
```

- `state`: необязательное значение:
  - `trimmed`
  - `eased`
  - `luffing`
- `angle_deg`: необязательное число.
- Для нетиповых положений вроде `goosewing` лучше задавать `angle_deg` явно, а не полагаться только на автоматический вывод угла.

Генакер:

```json
{
  "visible": true,
  "state": "trimmed"
}
```

- `visible`: обязательный boolean, если объект `gennaker` вообще указан.
- `state`: необязательное значение из того же списка.
- `angle_deg`: необязательное число.

## Объект `mark`

Используется в задачах у знака:

```json
{
  "x": 580,
  "y": 220,
  "zone_radius": 80
}
```

- `x`: число.
- `y`: число.
- `zone_radius`: число.

## Объект `overlays`

Используется для визуальных подсказок в карточках про правила:

```json
{
  "keep_clear_boat_id": "bravo",
  "windward_boat_id": "alpha",
  "leeward_boat_id": "bravo",
  "overlap_pairs": [
    {
      "a": "alpha",
      "b": "bravo"
    }
  ]
}
```

Поддерживаются поля:

- `keep_clear_boat_id`
- `windward_boat_id`
- `leeward_boat_id`
- `overlap_pairs`: массив объектов `{ "a": "...", "b": "..." }`

## Объект `answer_scene`

`answer_scene` нужен, когда после открытия ответа надо показать сцену не в текущем состоянии вопроса, а в отдельном итоговом состоянии.

Формат такой же, как у обычной сцены, но без вложенных `answer_scene` и `expected_answer`.

На практике:

- в `directional` часто показывает лодку уже на правильном курсе;
- в `term_definition` может скрывать правильную подсветку до открытия ответа;
- в `manoeuvre` показывает итог манёвра;
- в `right_of_way` часто содержит подсветку лодки через `highlight: "answer"`.

## Объект `expected_answer`

Поддерживаются 4 варианта.

### 1. Поворот лодки: `rotate_heading`

```json
{
  "type": "rotate_heading",
  "boat_id": "alpha",
  "heading_deg": 120,
  "tolerance_deg": 18
}
```

- `boat_id`: какую лодку вращает пользователь.
- `heading_deg`: правильный курс.
- `tolerance_deg`: допустимая погрешность.

Используется для `directional`.

### 2. Выбор варианта: `choose_option`

```json
{
  "type": "choose_option",
  "options": [
    { "id": "sheet-in", "label": "Выбрать грот-шкот" },
    { "id": "ease", "label": "Потравить грот-шкот" }
  ],
  "correct_option_id": "sheet-in"
}
```

- `options`: непустой массив вариантов.
- `options[].id`: внутренний идентификатор.
- `options[].label`: текст кнопки.
- `correct_option_id`: `id` правильного варианта.

Используется для `trim`.

### 3. Выбор лодки: `select_boat`

```json
{
  "type": "select_boat",
  "correct_boat_id": "bravo"
}
```

- `correct_boat_id`: `id` правильной лодки.

Используется для `right_of_way`.

### 4. Показ шагов: `reveal_steps`

```json
{
  "type": "reveal_steps",
  "steps": [
    "Проверить обстановку",
    "Плавно переложить руль",
    "Добрать паруса на новом галсе"
  ]
}
```

- `steps`: массив строк в правильном порядке.

Используется для `manoeuvre`.

## Практические шаблоны по текущим файлам

### `terms.json`

- Колода для терминов, курсов и базовых рулевых действий.
- Сейчас содержит `term_definition`, `directional` и `concept`.
- Для `directional` обычно используются сразу и `answer_scene`, и `expected_answer.rotate_heading`.
- Для части `term_definition` тоже полезен `answer_scene`, если правильная лодка или состояние должны появляться только после reveal.
- В версии схемы `1` часть терминов всё ещё опирается в основном на текст: здесь нет отдельных примитивов для снастей, вымпельного ветра, тактических линий и похожих элементов.

### `manoeuvres.json`

- Колода для работы с парусами и последовательностей действий.
- Сейчас содержит `trim` и `manoeuvre`.
- Для `trim` используются `expected_answer.choose_option`.
- Для `manoeuvre` обычно используются `answer_scene` и `expected_answer.reveal_steps`.
- Если `answer_scene` не меняет картинку заметно, он остаётся лишь как текстовое сопровождение и не добавляет новой визуальной подсказки.

### `right-of-way.json`

- Колода для правил расхождения и ситуаций у знака.
- Сейчас содержит `right_of_way` и `concept`.
- Обычно использует `overlays`, `expected_answer.select_boat` и `answer_scene`.
- В question scene лучше не оставлять answer-bearing overlays, если они фактически выдают правильную лодку до reveal. Исключение допустимо только для отдельных legend-карточек, где сам overlay и есть предмет обучения.
- Для правил, завязанных на историю ситуации, статичная сцена ограничена. Карточки про получение права дороги, изменение курса, позднюю связанность и похожие случаи часто честнее делать как `concept` с `choose_option`, а не как чистый diagram-quiz.
- В задачах у знака дополнительно используется `mark`.

## Рекомендации по редактированию

- Лучше копировать существующую карточку того же `template_type`, чем собирать новую с нуля.
- Если добавляете новую встроенную колоду, мало просто положить новый JSON в папку `content`: нужно ещё обновить список `CONTENT_DECK_ORDER` в `backend/db/builtin_content.py`.
- Для новых карточек держите `slug` коротким, стабильным и URL-подобным.
- Не пропускайте `answer_scene` там, где финальная картинка должна заметно отличаться от сцены вопроса.
- В `right_of_way` лучше явно заполнять `overlays`, чтобы схема была понятнее ученику.
